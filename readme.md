Ты должен разработать веб-приложение для автоматизации создания, печати и учета сертификатов.

ВАЖНО: не используй конкретный стек, если он уже не задан в проекте. Сначала изучи текущий проект, структуру файлов, зависимости и архитектуру. Реализуй решение в рамках существующего проекта, не ломая текущую логику.

Дополнительное обязательное требование: для реализации интерфейса редактирования сертификата необходимо использовать какой-либо готовый фреймворк, библиотеку или плагин для работы с документами/текстом, похожий по поведению на Word (например, rich text editor или document editor). Это нужно для удобного отображения страницы A4, позиционирования текста и работы с редактируемыми полями.

Контекст задачи:

У компании есть официальный заранее напечатанный бланк сертификата. На самом бланке уже есть шапка, логотипы, фоновые элементы, номер сбоку и другие декоративные/служебные элементы. В приложении нужно работать только с черным текстом сертификата, который печатается поверх готового бумажного бланка.

Сейчас сотрудники заполняют сертификат вручную в Microsoft Word или Microsoft Publisher, затем печатают его на готовый бланк, а после этого вручную переносят данные в Excel-реестр. Нужно заменить этот процесс одной программой.

К задаче прикреплены:

1. фотографии шапки и визуального вида сертификата;
2. Word-документ сертификата как пример шаблона;
3. пример расположения текста на сертификате.

Нужно внимательно изучить прикрепленные материалы и воспроизвести в приложении только тот текст, который должен печататься на бланке.

Основной workflow приложения:

1. Пользователь открывает приложение.
2. На главной странице он видит два основных раздела:

   * «Новый сертификат»
   * «Реестр»
3. По умолчанию должен открываться раздел «Новый сертификат».
4. В разделе «Новый сертификат» должен отображаться шаблон сертификата в виде страницы формата A4.
5. Визуально это должно быть похоже на текстовый редактор уровня Word: пользователь должен видеть страницу, текст, отступы, позиционирование и поля для заполнения.
6. В шаблоне сертификата уже должен быть заранее вставлен весь постоянный текст.
7. Места, которые пользователь должен заполнить, должны быть визуально выделены как редактируемые поля, желательно с подчеркиванием или другим аккуратным обозначением.
8. Пользователь заполняет только нужные поля внутри сертификата.
9. После заполнения пользователь нажимает кнопку «Сохранить в реестр».
10. Данные из конкретных полей сертификата автоматически переносятся в соответствующие колонки реестра.
11. После сохранения сертификат должен остаться доступен для просмотра/редактирования, если это необходимо.
12. Также должна быть кнопка «Печать сертификата».
13. При печати сертификат должен печататься один в один так, как он выглядит на экране, с тем же расположением текста на странице A4.
14. Печать должна быть рассчитана именно на уже готовый бумажный бланк, поэтому фон, шапку, логотипы и декоративные элементы печатать не нужно, если они уже есть на физическом бланке.
15. Нужно обеспечить максимально точное совпадение координат текста при печати.

Раздел «Новый сертификат»:

Нужно создать интерфейс редактирования сертификата.

Требования:

* страница формата A4;
* точные поля, отступы и позиционирование текста;
* весь постоянный текст сертификата уже должен быть вставлен;
* изменяемые поля должны быть редактируемыми;
* пользователь не должен случайно ломать структуру сертификата;
* желательно ограничить редактирование только тех мест, которые должны заполняться;
* текст должен выглядеть максимально близко к Word-документу;
* обязательно использовать библиотеку/фреймворк для редактирования текста (rich text editor или document editor), чтобы обеспечить поведение, похожее на Word;
* должна быть возможность сохранить данные;
* должна быть возможность распечатать сертификат.

Важно: не нужно делать полноценный Word со всеми возможными функциями. Главное — точное отображение шаблона, удобное заполнение полей и точная печать.

Раздел «Реестр»:

Нужно создать реестр сертификатов в виде Excel-подобной таблицы.

Обязательное требование: использовать AG Grid Community.

Реестр должен поддерживать:

* отображение всех сохраненных сертификатов;
* колонки, соответствующие данным из сертификата;
* фильтрацию;
* сортировку;
* поиск;
* редактирование ячеек, если это безопасно;
* изменение ширины колонок;
* закрепление/перемещение колонок, если возможно;
* экспорт данных, если возможно в рамках бесплатных возможностей;
* подсчет или агрегирование значений, если это нужно для числовых колонок;
* удобную работу как в Excel.

Данные реестра должны храниться в Supabase.

Требования к Supabase:

* создать/использовать таблицу для сертификатов;
* каждая запись в реестре должна соответствовать одному сертификату;
* при сохранении сертификата данные должны записываться в Supabase;
* при открытии реестра данные должны загружаться из Supabase;
* изменения в реестре должны синхронизироваться с базой данных;
* предусмотреть поля created_at и updated_at;
* предусмотреть уникальный идентификатор записи;
* не хранить Supabase service role key на клиенте;
* использовать только безопасные публичные ключи и переменные окружения.

Примерная структура данных сертификата:

Нужно самостоятельно определить точные поля на основе Word-документа и визуального шаблона. Каждое редактируемое место в сертификате должно соответствовать отдельному полю данных.

Например:

* номер сертификата;
* дата выдачи;
* наименование организации;
* наименование продукции;
* нормативный документ;
* срок действия;
* заявитель;
* изготовитель;
* страна;
* основание выдачи;
* дополнительные сведения;
* ответственное лицо;
* любые другие поля, которые реально есть в шаблоне.

Точные названия полей и колонок нужно взять из прикрепленного Word-документа и сертификата.

Кнопки и действия:

В разделе «Новый сертификат» должны быть кнопки:

* «Сохранить в реестр»
* «Печать сертификата»
* «Очистить форму»
* «Загрузить из реестра» или возможность открыть существующую запись для редактирования, если это логично

В разделе «Реестр» должны быть возможности:

* просмотреть все сертификаты;
* открыть выбранный сертификат;
* редактировать запись;
* удалить запись, если это предусмотрено;
* фильтровать и сортировать данные;
* экспортировать данные, если это возможно.

Печать:

Это критически важная часть.

Нужно реализовать отдельный print layout.

Требования:

* формат A4;
* точное позиционирование текста;
* печатать только нужный черный текст;
* не печатать интерфейс, кнопки, сетку, подсказки и лишние элементы;
* при печати скрывать все UI-элементы;
* использовать print CSS;
* настроить page size A4;
* убрать браузерные поля по максимуму;
* обеспечить совпадение с экраном;
* добавить инструкцию или комментарий в коде, где можно вручную подкорректировать координаты текста, если физическая печать немного смещается.

UX:

Интерфейс должен быть простым:

* сверху или сбоку навигация: «Новый сертификат» / «Реестр»;
* чистый рабочий экран;
* минимум лишних элементов;
* понятные кнопки;
* аккуратные уведомления об успешном сохранении, ошибке и печати;
* защита от случайной потери введенных данных.

Важные ограничения:

* не ломать существующий проект;
* не использовать платные библиотеки;
* AG Grid использовать только Community-версию;
* не использовать AG Grid Enterprise;
* не хранить секретные ключи в коде;
* не делать декоративные элементы сертификата, если они уже есть на физическом бланке;
* главный приоритет — точная печать и автоматический перенос данных в реестр.

Что нужно сделать по шагам:

1. Проанализировать текущий проект.
2. Найти подходящее место для страниц/компонентов.
3. Создать раздел «Новый сертификат».
4. Создать A4-шаблон сертификата на основе прикрепленного Word-документа.
5. Подключить и настроить библиотеку/редактор для работы с документом (rich text editor или аналог Word).
6. Сделать редактируемые поля внутри шаблона.
7. Создать структуру данных для этих полей.
8. Добавить кнопку сохранения в реестр.
9. Настроить Supabase для хранения записей.
10. Создать раздел «Реестр».
11. Подключить AG Grid Community.
12. Вывести данные из Supabase в AG Grid.
13. Настроить фильтры, сортировку, поиск и редактирование.
14. Добавить возможность открыть запись из реестра обратно в шаблоне сертификата.
15. Реализовать точную печать сертификата.
16. Проверить production build.
17. Подготовить проект к пушу в GitHub.
18. Убедиться, что проект может деплоиться на Vercel.

Критерии готовности:

* приложение открывается без ошибок;
* есть раздел «Новый сертификат»;
* есть раздел «Реестр»;
* шаблон сертификата визуально похож на Word-документ;
* используется библиотека/редактор для работы с документом;
* пользователь может заполнить поля;
* данные сохраняются в Supabase;
* данные появляются в AG Grid-реестре;
* реестр поддерживает Excel-подобные функции;
* сертификат можно распечатать;
* при печати выходит только нужный текст;
* production build проходит успешно;
* проект готов к GitHub и Vercel.
